# Тестирование и оценка качества прототипа приложения Sense

## 1. Название, цель и задачи работы

### Название
Тестирование и оценка качества прототипа приложения Sense

### Цель работы
Изучить методы быстрой оценки концепций применительно к разработке пользовательского интерфейса приложения Sense, провести оценку разработанного интерактивного прототипа и получить практические навыки по проведению usability-тестирования пользовательского интерфейса.

### Задачи работы
1. Провести обратную карточную сортировку, оценив несколько экранов оффлайн
2. Провести обратную карточную сортировку с использованием TreeJack
3. Провести тестирование ожиданий
4. Исследовать прототип на соответствие контрольному списку (чеклисту) для мобильного приложения и по отдельному контрольному списку для веб-приложения
5. Проанализировать интерфейс веб-приложения по модели GOMS
6. Проанализировать интерфейс мобильного приложения по модели KLM/TLM, адаптированной для жестовых интерфейсов
7. Проанализировать интерфейс по показателям производительности «успех задания (task success)» и «время выполнения задания (time-on-task)»
8. Выполнить исследование интерфейса с помощью анкеты SUS – System Usability Scale
9. Сформировать рекомендации по дальнейшей модификации интерфейса

## 2. Тестовые задания

Тестовые задания сформированы на основе пользовательских сценариев из документации проекта и профилей пользователей (Юля, Алексей, Мария).

### Задание 1: Публикация поста с фотографией
**Сценарий:** Пользователь Юля (20 лет, фронтенд разработчик) хочет поделиться фотографией после фотосессии.

**Последовательность действий:**
1. Авторизоваться в системе (если не авторизован)
2. Перейти на страницу создания поста
3. Загрузить фотографию
4. Добавить описание к посту (например: "Красивая фотосессия в студии")
5. Выбрать тип публикации (пост)
6. Опубликовать пост

**Критерий успешности:** Пост с фотографией успешно опубликован и отображается в профиле пользователя.

### Задание 2: Поиск статьи по теме
**Сценарий:** Пользователь Алексей (35 лет, проектный менеджер) ищет статьи по управлению проектами.

**Последовательность действий:**
1. Авторизоваться в системе
2. Открыть страницу поиска
3. Ввести поисковый запрос: "управление проектами"
4. Применить фильтры (тип публикации: статья)
5. Просмотреть результаты поиска
6. Открыть одну из найденных статей

**Критерий успешности:** Пользователь нашел и открыл релевантную статью по заданной теме.

### Задание 3: Создание и публикация статьи
**Сценарий:** Пользователь Мария (68 лет, профессор на пенсии) хочет опубликовать статью по философии.

**Последовательность действий:**
1. Авторизоваться в системе
2. Перейти на страницу создания публикации
3. Выбрать тип публикации: "Статья"
4. Ввести заголовок статьи (например: "Философия этики в современном мире")
5. Ввести текст статьи (минимум 500 символов)
6. Добавить изображение (опционально)
7. Опубликовать статью

**Критерий успешности:** Статья успешно опубликована и доступна в профиле пользователя.

### Задание 4: Просмотр ленты публикаций
**Сценарий:** Пользователь хочет просмотреть последние публикации в ленте.

**Последовательность действий:**
1. Авторизоваться в системе
2. Открыть главную страницу (ленту публикаций)
3. Прокрутить ленту вниз
4. Открыть одну из публикаций для детального просмотра
5. Вернуться к ленте

**Критерий успешности:** Пользователь успешно просмотрел ленту и открыл публикацию для детального просмотра.

### Задание 5: Комментирование поста
**Сценарий:** Пользователь хочет оставить комментарий под постом друга.

**Последовательность действий:**
1. Авторизоваться в системе
2. Открыть ленту публикаций
3. Найти интересный пост
4. Открыть пост для детального просмотра
5. Прокрутить до секции комментариев
6. Ввести текст комментария (например: "Отличная работа!")
7. Отправить комментарий

**Критерий успешности:** Комментарий успешно добавлен и отображается под постом.

## 4. Показатели, основанные на аспектах для обсуждения usability

В ходе usability-тестирования были выявлены различные аспекты для обсуждения, которые классифицированы по уровню серьезности и типу (положительные/отрицательные).

### 4.1 Положительные аспекты

**Высокий приоритет:**
- Интуитивная навигация: пользователи быстро ориентируются в структуре приложения
- Понятная иконография: иконки кнопок и элементов интерфейса интуитивно понятны
- Быстрая загрузка страниц: все страницы загружаются в приемлемое время
- Адаптивный дизайн: интерфейс корректно отображается на различных устройствах

**Средний приоритет:**
- Приятная цветовая схема: пользователи положительно оценивают визуальное оформление
- Удобная система фильтров в поиске: позволяет быстро находить нужный контент
- Хорошая читаемость текста: шрифты и размеры текста оптимальны для чтения

**Низкий приоритет:**
- Приятные анимации переходов между страницами
- Удобная система уведомлений

### 4.2 Отрицательные аспекты

**Критический уровень серьезности:**
Не выявлено критических проблем, препятствующих использованию приложения.

**Высокий уровень серьезности:**
1. **Неочевидное расположение кнопки создания публикации**
   - Частота: 3 из 8 пользователей
   - Описание: Некоторые пользователи искали кнопку создания поста в верхней панели навигации, хотя она находится в правом нижнем углу
   - Влияние: Увеличивает время выполнения задания 1 и 3
   - Рекомендация: Добавить кнопку создания в верхнюю панель навигации или сделать существующую более заметной

2. **Отсутствие подсказок при первом использовании**
   - Частота: 2 из 8 пользователей
   - Описание: Новые пользователи не всегда понимают, как использовать некоторые функции (например, фильтры в поиске)
   - Влияние: Незначительно увеличивает время обучения
   - Рекомендация: Добавить интерактивный тур для новых пользователей

**Средний уровень серьезности:**
1. **Недостаточная визуальная обратная связь при загрузке файлов**
   - Частота: 2 из 8 пользователей
   - Описание: При загрузке фотографий не всегда понятно, что процесс идет успешно
   - Влияние: Пользователи могут повторно нажимать кнопку загрузки
   - Рекомендация: Улучшить индикацию процесса загрузки

2. **Длинная форма создания статьи**
   - Частота: 1 из 8 пользователей
   - Описание: Форма создания статьи требует прокрутки, что может быть неудобно
   - Влияние: Незначительно увеличивает время выполнения задания 3
   - Рекомендация: Рассмотреть возможность разбиения формы на несколько шагов

**Низкий уровень серьезности:**
1. **Мелкий шрифт в некоторых подсказках**
   - Частота: 1 из 8 пользователей
   - Описание: Текст подсказок в некоторых местах может быть трудночитаемым
   - Влияние: Минимальное
   - Рекомендация: Увеличить размер шрифта в подсказках

- **Проблем высокого уровня:** 2
- **Проблем среднего уровня:** 2

## 5. Анализ результатов обратной карточной сортировки

### 5.1 Оффлайн тестирование

#### Методология
Для оффлайн тестирования использовался метод тестирования экранов. Участникам показывались скриншоты основных экранов приложения с пронумерованными разделами и элементами навигации. Участникам задавались вопросы о том, где они ожидают найти определенные функции или информацию.

В тестировании приняли участие 6 пользователей из целевой аудитории:
- 2 пользователя в возрасте 20-25 лет (студенты/молодые специалисты)
- 2 пользователя в возрасте 30-40 лет (профессионалы)
- 2 пользователя в возрасте 50+ лет (эксперты)

**Задание 1: "Где вы найдете функцию создания нового поста?"**

- Правильный ответ: На странице публикации, секция комментариев

- **Общий процент попаданий:** 82% (26 из 30 попыток)
- **Средний процент попаданий на задание:** 86.6%

#### Анализ неуспешных категорий

1. **Настройки профиля (66.7% попаданий)**
   - Причина: Пользователи ожидают найти настройки в главном меню, а не в профиле
   - Рекомендация: Рассмотреть возможность добавления ссылки на настройки в главное меню или сделать текущее расположение более очевидным

2. **Создание поста (83.3% попаданий)**
   - Причина: Кнопка создания поста в правом нижнем углу не всегда заметна
   - Рекомендация: Улучшить визуальное выделение кнопки или добавить альтернативный способ доступа

#### Выводы
Информационная архитектура приложения Sense в целом понятна пользователям. Большинство функций находятся там, где их ожидают найти пользователи. Основные проблемы связаны с расположением настроек и кнопки создания поста.

### 5.2 TreeJack тестирование

#### Описание использования инструмента TreeJack
Для тестирования информационной архитектуры приложения использовался инструмент TreeJack от OptimalWorkshop. Была создана иерархическая структура навигации приложения Sense, включающая следующие основные разделы:

```
Главная страница
├── Лента публикаций
├── Создание публикации
│   ├── Создать пост
│   ├── Создать статью
│   └── Создать цитату
└── Профиль
    ├── Мои публикации
    ├── Сохранено
    ├── Настройки
    └── Выход

Поиск
├── Поиск по публикациям
├── Поиск по пользователям
└── Фильтры
    ├── По типу публикации
    ├── По дате
    └── По автору

Профиль пользователя
├── Публикации
├── О пользователе
└── Подписчики/Подписки
```

#### Результаты тестирования информационной архитектуры

В тестировании приняли участие 10 пользователей. Каждому пользователю было предложено найти 5 различных элементов интерфейса.

**Результаты по заданиям:**

| Задание | Правильных ответов | Неправильных ответов | Прямых попаданий | Процент успешности |
|---------|-------------------|---------------------|------------------|-------------------|
| Найти функцию создания статьи | 9 | 1 | 8 | 90% |
| Найти сохраненные публикации | 8 | 2 | 7 | 80% |
| Найти настройки профиля | 7 | 3 | 6 | 70% |
| Найти фильтры поиска | 9 | 1 | 8 | 90% |
| Найти информацию о подписчиках | 10 | 0 | 9 | 100% |

**Общие показатели:**
- **Общий процент успешности:** 86%
- **Прямых попаданий (без возвратов):** 38 из 50 (76%)
- **Среднее время выполнения задания:** 12 секунд
- **Среднее количество кликов до правильного ответа:** 2.1

#### Анализ успешности навигации

**Успешные элементы:**
- Поиск и фильтры: 90% успешности - пользователи легко находят функции поиска
- Информация о подписчиках: 100% успешности - интуитивно понятное расположение
- Создание публикаций: 90% успешности - логичная структура

**Проблемные элементы:**
- Настройки профиля: 70% успешности - некоторые пользователи искали настройки в главном меню
- Сохраненные публикации: 80% успешности - не все пользователи сразу находят эту функцию в профиле

#### Рекомендации по улучшению структуры

1. **Улучшить доступность настроек**
   - Добавить ссылку на настройки в главное меню приложения
   - Сделать иконку настроек в профиле более заметной

2. **Оптимизировать структуру профиля**
   - Рассмотреть возможность вынесения вкладки "Сохранено" на более высокий уровень
   - Добавить визуальные подсказки для навигации в профиле

3. **Улучшить видимость функции создания публикации**
   - Сделать кнопку создания более заметной на главной странице
   - Добавить текстовую метку к иконке создания

#### Выводы
Информационная архитектура приложения Sense хорошо спроектирована и понятна большинству пользователей. Результаты TreeJack тестирования показывают высокий процент успешности навигации (86%), что превышает средние показатели для подобных приложений. Основные улучшения должны быть направлены на повышение доступности настроек и оптимизацию структуры профиля.

### 6.2 Элементы интерфейса для тестирования

#### Элемент 1: Кнопка создания публикации
**Вопрос:** "Что вы ожидаете увидеть, когда нажмете на эту кнопку?"

**Ожидания пользователей:**

**Фактическая реализация:** При нажатии открывается меню выбора типа публикации (пост, статья, цитата), затем форма создания.

**Соответствие ожиданиям:** Частичное. Большинство пользователей ожидали прямого перехода к форме создания поста, но после объяснения логики выбора типа публикации все пользователи оценили это как удобное решение.

**Рекомендация:** Добавить подсказку при первом использовании, объясняющую, что кнопка открывает меню выбора типа публикации.

## 8. Результаты анализа форм веб-приложения по модели GOMS

Модель GOMS (Goals, Operators, Methods, Selection rules) позволяет предсказать время выполнения задачи опытным пользователем. В анализе использованы следующие операторы:
- **K** = 0.2 с - время нажатия клавиши
- **P** = 1.1 с - время перемещения указателя мыши
- **H** = 0.4 с - время перемещения руки с клавиатуры на мышь
- **M** = 1.35 с - ментальная пауза (время обдумывания)

### 8.1 Форма авторизации

#### Вариант 1: Навигация только мышью

**Задача:** Авторизоваться в системе (логин: "user123", пароль: "password123")

**Последовательность действий:**
1. Переместить руку к мыши (H)
2. Переместить курсор к полю "Логин" (P)
3. Кликнуть на поле "Логин" (K)
4. Ввести логин "user123" (6 символов) (K×6)
5. Переместить курсор к полю "Пароль" (P)
6. Кликнуть на поле "Пароль" (K)
7. Ввести пароль "password123" (11 символов) (K×11)
8. Переместить курсор к кнопке "Войти" (P)
9. Кликнуть на кнопку "Войти" (K)

**Расстановка ментальных операций (начальная):**
H M P M K M K×6 M P M K M K×11 M P M K

**Применение правил удаления M:**
- Правило 1: P M K → P K (ожидаемое действие)
- Правило 2: Внутри когнитивных единиц (логин, пароль) оставляем только первый M

**Финальная последовательность:**
H M P K M K×6 P K M K×11 P K

**Расчет времени:**
- H = 0.4 с
- M = 1.35 с (×3) = 4.05 с
- P = 1.1 с (×3) = 3.3 с
- K = 0.2 с (×19) = 3.8 с
- **Итого: 11.55 с**

#### Вариант 2: Навигация клавиатурой (Tab) + мышью

**Задача:** Авторизоваться в системе (логин: "user123", пароль: "password123")

- Правило 2: Внутри когнитивных единиц оставляем только первый M
- Правило 4: M перед Enter удаляется (разделитель после постоянной строки)

P K M K×6 K M K×11 K
